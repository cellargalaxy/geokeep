## 1. 目标与范围

### 1.1 核心目标
- 自建、单实例、单租户为主的「个人位置数据保管 + 可视化」服务，可看作 dawarich 的最小子集。
- **不做**订阅、付费分层、Fog of War、Heatmap、Globe 3D、Trips 自动识别、Visits/Places 自动检测、Immich/PhotoPrism 集成、Strava 联动、反向地理编码（Reverse Geocoding）等增值功能。
- 必须能与 dawarich 双向迁移；必须接收 OwnTracks / Overland 实时上报；必须在 OpenStreetMap 上展示轨迹。
- 全部操作（初始化、导入导出、备份恢复、api_key 与分享链接管理）均在 **Web 页面**完成，**不提供 CLI**；前端按**多页面**架构组织并预留扩展位，同时适配**电脑端与手机端（响应式）**。
- 提供**免登录只读分享链接**（URL 携带 token）供对外分享；服务端**代理并缓存 OSM 瓦片**以降低对公共瓦片接口的重复请求（§5.4）。

### 1.2 范围边界（MVP 圈定）

| 范围 | 包含 | 不包含（明确剔除） |
|---|---|---|
| 用户体系 | 单管理员账号 + 单一 API Key（Web 内生成/轮换）+ 免登录只读分享 token | 多租户 / OAuth / SSO / TOTP |
| 上报 | OwnTracks HTTP、Overland HTTP | MQTT、GPSLogger、Home Assistant 推送（V2 再议） |
| 导入 | dawarich v1 `data.json`、v2 JSONL archive、OwnTracks `.rec`、GPX、GeoJSON（不限文件大小） | Google Takeout、KML/KMZ、Strava、Immich/PhotoPrism |
| 导出 | dawarich v2 JSONL archive、GPX、GeoJSON、OwnTracks JSON（行式；不限文件大小） | KML/KMZ |
| 可视化 | 多页面 Web（响应式，适配电脑/手机）：地图（点+轨迹+回放）、收录点列表表格查询、导入/导出/备份/设置/分享管理 | 热力图、Fog of War、3D 地球、统计图表 |
| 瓦片 | 服务端代理并缓存 OSM 瓦片（前端经服务端取图，§5.4） | 自建完整 tile server、矢量瓦片 |
| 持久化 | SQLite（单文件） | Postgres / PostGIS / Redis / Sidekiq |
| 备份 | 页面导出/上传备份文件 + 服务停机后直接复制/替换 `geokeep.db` | 自动定时备份、增量备份、对象存储集成 |

### 1.3 关键约束
- **单文件持久化**：所有业务数据落在同一 `geokeep.db`（SQLite，WAL 模式），用户拷走该文件即等价完整备份；导入文件 + 触发恢复 = 还原。
- **改动最小 / 复用优先**：协议字段命名、上报 URL 路径、导出包结构尽可能对齐 dawarich 与官方客户端，避免在客户端做任何 fork/定制。
- **轻量部署**：单二进制 + 嵌入静态前端（地图页）；零外部依赖（无 Redis / 反代均可独跑）。
- **数据保真**：原始上报 payload 必须**全量留存**到 `points.raw_data`（JSON 文本），任何解析失败的字段都不能丢，对应 dawarich `points.raw_data jsonb`。
- **全页面化 / 无 CLI**：初始化向导、导入导出、备份恢复、api_key 与分享链接管理全部在 Web 页面完成，Docker 用户无须进容器执行命令。
- **多页面 + 响应式前端**：前端按多页面（地图 / 点列表 / 导入 / 导出 / 备份 / 设置 / 分享管理 / 登录·初始化）组织，预留后续扩展页面；同一套页面响应式适配电脑与手机（窄屏侧栏折叠）。
- **瓦片缓存**：前端不直连公共 OSM 瓦片，统一经服务端 `GET .../tiles/{z}/{x}/{y}.png` 代理并落盘缓存（详见 §5.4）。
- **导入/导出文件不限大小**：上传/下载全程流式，不设固定大小上限（仅受磁盘约束）；上报通道仍保留请求体上限（§3.3）。

## 2. 数据模型（SQLite）

> 字段命名与 dawarich `db/schema.rb` 对齐，便于双向迁移；类型按 SQLite affinity 调整。所有时间统一为 Unix 秒（与 OwnTracks `tst` 一致）。

剔除的 dawarich 字段及理由（**实测点字段见 §13.2**）：
- 实测 v2 点对象中需剔除的列：`lonlat (PostGIS WKB)`、`external_track_id`、`reverse_geocoded_at`、`city`、`country`、`country_info`、`geodata`、`import_reference`、`ping`、`mode`、`created_at/updated_at(dawarich 侧)` —— **均依赖未做的能力（PostGIS、反向地理编码、Tracks、Imports 关联）或语义未明**，MVP 不建独立列；导入时**整体随 `raw_data` 留存**（不丢数据），导出时不还原（§4.3 丢失列白名单）。
- 注：早期推测的 `country_id / track_id / visit_id / raw_data_archive_id / raw_data_archived / anomaly / country_name` **未出现在实测 v2 点对象中**，以 §13.2 实测键集为准。

索引：
- `idx_points_user_timestamp(user_id, timestamp)` —— 地图按时间窗查询主路径
- `idx_points_user_device_timestamp(user_id, device_id, timestamp)`
- 唯一性约束：`UNIQUE(user_id, latitude, longitude, timestamp)` —— **对齐 dawarich 「坐标 + 时间戳」去重规则**（来源：search 摘要「validated for uniqueness by its coordinates and timestamp」）。

## 3. 接收上报 API（与 dawarich 客户端兼容）

### 3.1 OwnTracks

- **路由**：`POST /api/v1/owntracks/points?api_key={api_key}`（与 dawarich 路径一致）
- **认证（任一即可）**：
  - 查询串 `api_key=...`
  - HTTP Basic（用户名/密码映射到 `users.email / api_key`，待确认是否启用）
  - `X-Limit-U` / `X-Limit-D` Header 仅用于解析 user/device 名称，**不作为鉴权**（依据：OwnTracks HTTP 文档）
- **请求体**：单个 JSON 对象，需路由 `_type`：
  - `_type=location` → 落 `points`
  - `_type=transition / waypoint / status / cmd / lwt` → **MVP 仅完整存档到 `raw_data`，不建模为业务行为**，返回 200
- **响应体**：固定 `[]`（依据：OwnTracks 文档：「typically return an empty JSON payload array `[]`」）；MVP 不下发好友/cmd。
- **去重**：触发 `UNIQUE(user_id, latitude, longitude, timestamp)` 冲突时静默忽略，并在响应日志记 `dup=1`。
- **错误**：鉴权失败 401（**返回空数组而非错误体**，待确认；OwnTracks 文档未明确，但客户端只关心 2xx）。

### 3.2 Overland

- **路由**：`POST /api/v1/overland/batches?api_key={api_key}`（与 dawarich 路径一致）
- **认证**：
  - `Authorization: Bearer {api_key}`（Overland 原生配置项）
  - 兜底支持 `?api_key=...`
- **请求体**：`{"locations":[Feature...], "current":{...}?, "trip":{...}?}`
  - 每个 Feature 拆为一行 `points`
  - `current / trip` 字段整体落 `raw_data.overland_extra`（不丢，但不建模）
- **响应体**：必须 `{"result":"ok"}`（否则客户端会本地堆积重传，依据：Overland README）。
- **字段转换备注**：
  - `properties.timestamp`（ISO8601）→ epoch 秒
  - `properties.speed`（m/s）→ km/h 时需在 `points.velocity` 中以**字符串**形式（与 dawarich 一致）；推荐**保留 m/s 原值**于 `raw_data`，`velocity` 写 km/h 整数字符串，**取舍**：兼容 dawarich 类型优先于单位忠实；该约定记入字段注释。
  - `properties.battery_level`（0–1）→ ×100 取整。
  - `properties.battery_state` 字符串 → 0/1/2/3 映射（待确认与 OwnTracks `bs` 完全一致）。

### 3.3 通用要求
- 请求体大小上限：仅**上报通道**默认 5 MB（Overland 一次 `locations_in_payload` 可达 1000，按 1KB/Feature 估算）；**导入/导出**通道不设大小上限（§4.0）。
- 同步写：MVP 不引入异步队列；写库失败直接返回非 2xx，由客户端重传。
- 限流：MVP 不做；只在反代层建议（文档化）。

## 4. 导入 / 导出（dawarich 兼容）

### 4.0 文件大小、格式来源与 dawarich 导出操作

- **不限文件大小**：导入上传与导出下载全程流式（v2 JSONL 逐行、v1 `data.json` 走 `json.Decoder` token 流、归档用 `archive/tar|zip` 顺序读），不设固定大小上限，仅受磁盘空间约束（§3.3 的 5MB 仅作用于上报通道）。
- **「兼容格式来自什么文档」**：导入清单源自 `https://dawarich.app/docs/features/imports/`；「导出为 OwnTracks JSON 同构」源自 `https://dawarich.app/docs/getting-started/export-your-data/`「structured the same way as the OwnTracks JSON format」；v1 `data.json` 与 v2 JSONL 月份分片差异源自 Release 1.3.0 / Patreon 0.13.0 公告（均见 §0）。
- **「如何在 dawarich 导出全量点数据」**（两条官方路径，供用户下载文件交给 geokeep 参考/导入）：
  1. **按时间窗导出点**：dawarich → **Points** 页 → 选时间范围 → 点 **Search** → 点 **Export**；任务后台异步生成，到 **Exports** 页下载；产物为 **OwnTracks JSON 同构**的 JSON（依据：export-your-data 文档）。
  2. **全量迁移归档**：dawarich → **Account Settings → Export My Data** 产出含 `points` 的归档（v1 `data.json` 或 v2 JSONL 月份分片）；对端用 **Account Settings → Import My Data** 回灌（依据：imports 文档「Export your data ... Import My Data」迁移闭环）。
  - geokeep 导入端对上述产物**两者皆收**（§4.1）；用户取得文件后在 geokeep「导入」页上传即可。
  - **待确认**：v2 归档内部精确目录/manifest 命名公开文档未列全，需实施期用一份真实 archive 固化为 fixture（§12（3））。

### 4.1 导入格式清单（MVP）

| 格式 | 来源 | 处理策略 |
|---|---|---|
| dawarich v2 archive（**首选**） | 用户 UI 上传 zip | 解压 → 优先读根级 `manifest.json` 的 `files.points` → 流式逐行读 **`points/<YYYY>/<YYYY-MM>.jsonl`**（**实测布局见 §13**，年子目录 + 年-月文件），按 §13.2 字段映射写入；其余实体文件（`stats/tracks/digests/tags/...`）**整体跳过**并写 `imports.error_message` 标注（详见 §13.4） |
| dawarich v1 archive | 用户 UI 上传 `data.json` | 流式 JSON（避免 OOM），逐对象映射 |
| OwnTracks `.rec` | 用户 UI 上传 | 每行 `2024-01-01T00:00:00Z\t*\t{json}`，提取 JSON 后按 OwnTracks 字段映射（与上报路径共用 mapper） |
| GPX | 用户 UI 上传 | 解析 `<trkpt lat lon><time/><ele/>`；缺失的 acc/vel 全部留空 |
| GeoJSON | 用户 UI 上传 | 仅支持 `FeatureCollection<Point>` 与 `Feature<LineString>`；`LineString` 需配对 `properties.coordTimes`，否则**整段拒绝**（不脑补时间戳） |

去重：所有导入走 `UNIQUE(user_id, lat, lon, ts)` —— 与 dawarich 一致。
进度：`imports` 表持久化 `raw_points / processed / doubles / status`；前端轮询。

### 4.2 导出格式清单（MVP）

| 格式 | 用途 | 内容 |
|---|---|---|
| **dawarich v2 archive**（**首选**） | 双向迁移核心 | 根级 `manifest.json` + **`points/<YYYY>/<YYYY-MM>.jsonl`**（严格复刻实测 v2 布局，§13.4）；其它实体文件不产出 |
| GPX | 通用 GPS 工具 | 单个 `<trk>`，按设备拆 `<trkseg>` |
| GeoJSON | 通用 GIS 工具 | `FeatureCollection<Point>`，`properties` 含全部已存字段 |
| OwnTracks JSON | 重新喂给 OwnTracks Recorder 等 | 每行一个 `_type=location` 对象（与上报 payload 同构） |

> **冲突点**：dawarich 同时有 v1 `data.json`（全量单 JSON）与 v2 JSONL 月份分片两种 archive；官方导入端**两者皆受**。geokeep MVP **只产出 v2**（更省内存、官方未来主推），但**进口端两者皆收**。
> **依据**：「new format streaming data to avoid memory issues while remaining backward-compatible with v1 archives」。

### 4.3 反向兼容（geokeep → dawarich）测试用例
- 使用 dawarich 自有 demo 数据导出 v2 → 导入 geokeep → 再从 geokeep 导出 v2 → 喂回 dawarich，比较 dawarich 内 `points.count` 与坐标-时间集合，**容忍丢失字段集**：`country / city / geodata / visit_id / track_id / external_track_id` 等 MVP 未持有的列（这些会变 NULL，由 dawarich 重新做反向地理编码后回填）。

## 5. 前端页面（多页面 / 响应式 / OpenStreetMap）

### 5.1 页面清单（多页面，响应式适配电脑/手机）
- **登录 / 初始化向导**：首次 `UserCount()==0` 展示初始化（注册首个管理员，一次性返回 api_key）；否则展示登录。
- **地图页**：Leaflet + OSM（经服务端瓦片缓存，§5.4）；左栏时间窗（默认今天）、设备多选、抽样阈值、播放控制；主区域点（`L.circleMarker`）+ 折线（`L.polyline`，按时间排序）+ 按时间步长回放。
- **收录点列表页**：表格查询收录的点（时间 / 设备 / 经纬度 / 海拔 / 速度 / 精度等列），支持时间窗 + 设备过滤 + 分页 + 排序（接口 §5.2）。
- **导入页 / 导出页**：创建任务、轮询进度、下载导出物。
- **备份页**：导出备份文件 / 上传备份恢复（§6）。
- **设置页**：显示 api_key 尾 4 位 + 生成/轮换（§7.6）；**客户端接入配置生成器**（OwnTracks / Overland / GPSLogger 一键复制，§13.6）；分享链接管理（创建/停用/删除，§5.5）。
- 不做：热力图、聚类、Fog of War、3D 视角。
> 前端按**多页面**架构组织并预留扩展位（未来新增功能即新增页面，§1.3）；同一套页面**响应式**适配电脑端与手机端（窄屏侧栏折叠为抽屉）。

### 5.2 后端查询接口（含设备/统计接口功能说明）
- `GET /api/v1/points?from=<epoch>&to=<epoch>&device_id=&limit=&sample=`：**地图页**按时间窗拉点；`sample` 等步长抽样（每 N 个点取 1 个）用于长窗预览，服务端实现避免前端拉全量。
- `GET /api/v1/points/list?from=&to=&device_id=&page=&size=&order=`：**收录点列表页**专用，分页 + 排序返回表格行；与地图接口分离以控制 payload。
- `GET /api/v1/devices` —— **设备列表接口功能**：返回当前用户的设备清单及每台设备的派生统计 `id / name / source / last_seen（该设备最后一个点的时间戳）/ point_count（点总数）`。用途：地图页/列表页的**设备过滤分组**与设备概览。设备由上报时按 OwnTracks `tid` / Overland `device_id` upsert 生成（§2.2）。
- `GET /api/v1/stats/summary?from=&to=&device_id=` —— **统计接口功能**：返回所选时间窗（可叠加设备过滤）内的汇总指标 `count`（点数）与 `distance_m`（经纬度球面 Haversine 现算的累计里程，无缓存表）。用途：页面顶部概览数字；**不做**复杂统计图表（§1.1）。

### 5.3 鉴权
- 前端登录 → Cookie Session（HttpOnly，SameSite=Lax）。
- 上报接口走 `api_key`（与会话隔离）。
- 分享链接走 URL `token` → 仅放行只读查询（points / points/list / devices / stats / tiles），命中 `requireShareReadonly`，拒绝一切写/管理接口（§5.5）。
- **三类凭证（Session / api_key / 分享 token）是相互隔离的三套方案，不共用**（对比表与理由见 §13.5）。

### 5.4 OSM 瓦片代理与缓存（新增）
- **动机**：前端不直连 `tile.openstreetmap.org`，避免对公共瓦片接口重复请求并触碰 OSM 用量政策（署名 / 合规 UA / 禁批量预取，见 §0）。
- **接口**：`GET /api/v1/tiles/{z}/{x}/{y}.png`（地图页与分享页均经此取图）。
- **缓存机制（设计）**：
  - 落盘缓存：`data/tiles/{z}/{x}/{y}.png`；命中直接回源文件并带 `Cache-Control` / `ETag`。
  - 未命中：服务端向上游 `GEOKEEP_OSM_TILE_URL`（默认 `https://tile.openstreetmap.org/{z}/{x}/{y}.png`）拉取，带合规 `User-Agent`（含联系方式），成功后写盘再返回。
  - 失效：按 `mtime` + TTL（默认 30 天，`GEOKEEP_TILE_TTL` 可配）过期回源刷新；磁盘超 `GEOKEEP_TILE_CACHE_MB` 上限时按 LRU 清理。
  - 合规：限制 `z` 范围（如 0–19）、对上游设并发上限与失败退避，避免被判定为批量抓取。
- **取舍**：仅缓存栅格瓦片，不内置完整 tile server；`data/tiles/` 可丢弃（仅触发重新回源），不计入业务备份。

### 5.5 免登录只读分享链接（新增）
- **形式**：URL 携带 token，形如 `/{base}/share/{token}`；无须账号密码即可访问对应只读视图。
- **权限**：**仅只读** —— 放行 `GET points / points/list / devices / stats/summary / tiles`；拒绝任何写操作、导入导出、备份恢复、设置与 token 管理接口。
- **过滤范围**：token 可绑定 `device_ids / from / to`（§2.6 `scope`），分享对象只能看到被授权子集。
- **管理页**：登录后在「设置 / 分享管理」页管理链接：创建（返回完整 URL，一次性展示）、命名、启用/停用（停用即失效）、设置过期、删除、查看 `last_used_at`。
- **接口**：见 §7.7（`/api/v1/share-links` CRUD + `/api/v1/share/{token}/...` 只读访问入口）。

## 6. 备份与恢复（基于 SQLite 文件）

### 6.1 设计原则
- **唯一真理源**：业务数据 100% 在 `geokeep.db`；导入产物（原始上传文件）放 `data/imports/<id>/`，导出产物放 `data/exports/<id>/`，**这两个目录可丢失**（仅影响重新下载历史导出包，不影响业务数据）。
- 启用 SQLite **WAL** 模式 + `synchronous=NORMAL`，保证拷贝时崩溃一致性。

### 6.2 备份方式（仅两种，**不做自动定时**）
1. **页面导出备份**：登录后在「备份」页点击导出，服务端 `GET /api/v1/backup` 触发 `VACUUM INTO` 生成一致性快照并流式下载 `geokeep-YYYYMMDD-HHMMSS.db`，避免直接 copy WAL/SHM 不一致。
2. **离线复制**：服务停机后直接复制运行目录的 `geokeep.db` 文件。
> 明确**不做**内置 cron 自动定时备份（用户需求：仅保留页面导出 + 离线复制两种方式）。

### 6.3 恢复方式（仅两种，**不做 CLI**）
1. **页面上传恢复**：登录后在「备份」页上传 `.db` 文件（`POST /api/v1/restore`，仅管理员）；服务端校验 SQLite 头、`PRAGMA integrity_check`、`PRAGMA user_version` 不高于当前，落标记后切换并提示重启 / 强制登出。
2. **离线替换**：服务停机后直接用备份文件替换 `geokeep.db`，再启动服务。
> 明确**不提供** `geokeep restore` CLI 子命令；一切恢复在页面完成（Docker 用户无须进容器）。

### 6.4 Schema 演进
- 使用「迁移版本号 + 顺序 SQL 脚本」（启动期自动迁移）
- 备份文件内必须能识别版本（`PRAGMA user_version`）；恢复时若版本**低于**当前，自动走前向迁移；**高于**当前直接拒绝（避免回滚损坏）。

## 8. 非功能性约束

| 项 | 目标 |
|---|---|
| 写入吞吐 | 个人最多 2–3 设备、读写要求不高；目标 ≥ 50 points/s 即足够（单实例 SQLite WAL） |
| 查询响应 | 30 天窗口 ≤ 200k 点，`?sample=` 后 ≤ 300ms 返回 |
| 内存 | 导入/导出不限大小，故**必须流式**：v2 JSONL 逐行、v1 `data.json` 用 `json.Decoder` token 流；常驻峰值 < 256 MB |
| 并发 | SQLite WAL 单写多读；导入/导出走单独 goroutine + 串行写锁；MVP 不引入外部队列 |
| 部署 | 单 Go 二进制 + 同目录 `data/`；Docker 镜像可选 |
| 时区 | 数据库内**统一 UTC epoch**；前端按浏览器时区显示 |
| 国际化 | 后端只输出英文/中文双语错误码；UI 文案中文优先 |

## 9. 方案取舍（按 prompt「冲突时列出」要求）

| 议题 | 取舍 | 依据 |
|---|---|---|
| Postgres+PostGIS vs SQLite | 选 SQLite，弃 PostGIS | 用户硬性要求；MVP 无空间索引需求（按 user_id+timestamp 主路径即可） |
| 引入异步队列（Sidekiq 类比）vs 同步处理 | 选同步 + 后台 goroutine | dawarich 用 Sidekiq 是因为多任务并发 + 反向地理编码慢；MVP 都不做 |
| `points.velocity` 类型 string vs number | 选 string | 对齐 dawarich `velocity string`，便于 archive 互转 |
| Overland speed m/s vs km/h 落库 | `velocity` 落 km/h 字符串（兼容 dawarich）、`raw_data` 留原 m/s | 兼容性 > 单位忠实，不丢原值 |
| 是否拉独立 `devices` 表 | 拉 | dawarich 平铺，但 geokeep 需要前端按设备过滤；不影响导出兼容（仍写回 `topic/tracker_id`） |
| v1 vs v2 archive 产出 | 出 v2、收 v1+v2 | v2 流式更省内存；v1 仍是大量历史用户基线 |
| Trips/Visits/Places 是否建模 | 不建模 | 用户「核心功能」未提，dawarich 这部分依赖反向地理编码与复杂检测，超出 MVP |
| 反向地理编码 | 不做 | 同上；导入的 city/country 字段全部丢入 raw_data |
| 鉴权方式 | Session（Web）+ api_key（上报）+ Share token（免登录只读） | OwnTracks/Overland 客户端均原生支持 token-in-URL 或 Bearer；分享场景只读隔离 |
| 自动定时备份 vs 仅页面/离线 | 仅页面导出 + 离线复制/替换 | 用户明确不要 cron 自动备份 |
| CLI vs 全页面化 | 全页面化（含初始化向导 / 上传恢复）| 用户要求一切操作在页面完成 |
| 单页 SPA vs 多页面 | 多页面 + 响应式 | 用户要求预留更多页面扩展、适配电脑/手机 |
| 瓦片直连 vs 服务端缓存代理 | 服务端缓存代理 | 降低公共 OSM 重复请求、便于署名/UA 合规 |
| 导入导出大小上限 | 不限（全程流式）| 用户要求不限制 |
| 免登录分享 | URL token + 只读 + 管理页 | 用户要求一种无须登录的只读分享形式 |
| 密码哈希 bcrypt vs argon2id | argon2id | OWASP 首选、无 bcrypt 72 字节限制（§12（1））|

## 10. 实施步骤建议（仅作为后续 prompt 排程参考，本次不实施）

1. **数据层骨架**：SQLite schema 迁移 + Go struct + `repository` 层；编写 UNIQUE 去重单测。
2. **上报通道**：OwnTracks / Overland handler + 字段 mapper + raw_data 留存单测（构造官方 README 样例对拍）。
3. **导入器**：先 dawarich v2 → v1 → OwnTracks `.rec` → GPX → GeoJSON；每个独立 streaming 解析器。
4. **导出器**：v2 archive → GPX → GeoJSON → OwnTracks JSON；与对应导入器做 round-trip 单测。
5. **查询 API + 瓦片缓存**：`points / points/list / devices / stats/summary` + `tiles/{z}/{x}/{y}.png` 代理缓存（§5.4）。
6. **多页面前端（响应式）**：地图（时间窗/设备过滤/回放）+ 收录点列表表格 + 导入/导出/备份/设置页；嵌入二进制。
7. **api_key 与分享链接**：`api-key/rotate` + `share-links` CRUD + `/share/{token}` 只读访问中间件（含越权/只读单测）。
8. **备份/恢复**：`VACUUM INTO` 导出接口 + 页面上传恢复（无 CLI、无 cron 自动定时）。

## 13. dawarich v2 归档真实结构（实测固化，权威依据）

> **来源**：真实导出文件 `user_data_export_20260613_133710.zip`（dawarich **Account Settings → Export My Data** 产出，`manifest.json` 自述 `dawarich_version: 1.8.1`、`format_version: 2`）。
> 该文件按 prompt 约定**用后即删**，故此节将其结构与 geokeep 处理方式**完整固化为依据**，替代原 §12（3）的「待确认」。

### 13.1 顶层布局
- **归档形式**：`.zip`（非 tar.gz）；条目均为 UTF-8 文本。
- **`manifest.json`（根级单文件，非 jsonl）**：
  ```json
  {
    "format_version": 2,
    "dawarich_version": "1.8.1",
    "exported_at": "2026-06-13T05:38:05Z",
    "counts": { "points": 85808, "tracks": 200, "stats": 21, "digests": 5,
                "imports": 18, "exports": 1, "notifications": 2617, "tags": 4,
                "areas": 0, "trips": 0, "visits": 0, "places": 0, "raw_data_archives": 0 },
    "files": {
      "points":  ["points/2023/2023-06.jsonl", "...", "points/2026/2026-06.jsonl"],
      "stats":   ["stats/2026/2026-06.jsonl", "..."],
      "tracks":  ["tracks/2026/2026-06.jsonl", "..."],
      "digests": ["digests/2024/2024.jsonl", "digests/2025/2025-10.jsonl", "..."],
      "visits":  []
    }
  }
  ```
  - `manifest.files` **只列时间分片实体**（points/visits/stats/tracks/digests），**不含**扁平实体文件。
- **时间分片实体**（路径 `<entity>/<YYYY>/<YYYY-MM>.jsonl`，每行一个 JSON 对象）：`points/`、`stats/`、`tracks/`、`digests/`。
  - `digests/` 同时含**年汇总** `digests/2024/2024.jsonl` 与**月** `digests/2025/2025-10.jsonl`。
- **扁平实体**（根级 `<entity>.jsonl`，单文件）：`areas.jsonl`、`places.jsonl`、`trips.jsonl`、`tags.jsonl`、`taggings.jsonl`、`imports.jsonl`、`exports.jsonl`、`notifications.jsonl`、`settings.jsonl`、`raw_data_archives.jsonl`。
- **空实体的不一致表现**（解析必须容错）：`areas/places/trips/taggings/raw_data_archives` 为 **0 字节文件**；`visits` **整个文件缺失**（仅在 manifest 里 `"visits": []`）。→ **解析器不得假设任何非 points 文件一定存在或非空**。

### 13.2 `points/*.jsonl` 单行真实字段（geokeep 映射依据）
实测点对象键（OwnTracks 实时上报样例 + Google Timeline 导入样例并集）：
```
battery_status, battery, timestamp, altitude, velocity, accuracy, ping,
tracker_id, topic, trigger, bssid, ssid, connection, vertical_accuracy, mode,
inrids, in_regions, raw_data, city, country, country_info, geodata,
reverse_geocoded_at, course, course_accuracy, external_track_id,
import_reference, created_at, updated_at, lonlat, longitude, latitude
```
关键事实：
- `timestamp` = **Unix 秒**（int）；`velocity` = **字符串**（如 `"0.0"`）；`trigger / connection / battery_status` = **整数或 null**。
- `raw_data` = **被转义的 JSON 字符串**（非嵌套对象），内含原始上报 payload + dawarich 追加字段（`action / api_key / controller / point`）。**注意：`raw_data` 内明文含上报 `api_key`** —— 导入时按原则全量留存，但**审计/分享导出场景需警惕泄露**。
- `inrids / in_regions`：OwnTracks 点为 **数组 `[]`**，导入点却为 **字符串 `"{}"`** —— 类型不稳定，解析需两者皆容忍。
- `import_reference`（`{name, source, created_at}`）、`country_info`（`{name, iso_a2, iso_a3}`）为嵌套对象，仅导入点出现。
- `lonlat` = PostGIS WKB 十六进制（如 `0101000020E61000...`）；`geodata`=`"{}"`。

geokeep 字段取舍（落地 §2.3）：
- **建模为独立列**：`timestamp / latitude / longitude / altitude / velocity / accuracy / vertical_accuracy / course / course_accuracy / battery / battery_status / connection / trigger / ssid / bssid / tracker_id / topic / inrids / in_regions / raw_data`。
- **整体并入 `raw_data`、不单独建列**（PostGIS / 反向地理编码 / dawarich 内部）：`lonlat / city / country / country_info / geodata / reverse_geocoded_at / external_track_id / import_reference / ping / mode / created_at / updated_at`。

### 13.4 geokeep 处理方式（导入 / 导出）
- **导入（dawarich v2 → geokeep）**：
  1. 打开 zip，**优先读 `manifest.json`** 取 `files.points` 列表；若无 manifest，则回退「遍历所有 `points/**/*.jsonl` 条目」。
  2. 对每个 points 文件**逐行** `json.Unmarshal` 单点 → 按 §13.2 映射写库；`raw_data` 原样保留。
  3. **跳过** points 以外的全部实体文件，并在 `imports.error_message` 追加 `skipped: areas,places,trips,visits,stats,tracks,digests,tags,taggings,imports,exports,notifications,settings,raw_data_archives`。
  4. 容错：非 points 文件可能 0 字节或缺失；`inrids/in_regions` 数组或 `"{}"` 皆接受；`trigger/connection/battery_status` 直接存整数原值（不再做字符串映射）。
  5. 若 zip 内无任何 `points/**/*.jsonl`，返回 422 `unrecognized archive layout`。
- **导出（geokeep → dawarich 可导入的 v2）**：
  1. 产 `.zip`，写**根级 `manifest.json`**（`format_version:2`、`dawarich_version` 填兼容标识、`exported_at`、`counts`、`files`）。
  2. 按月写 `points/<YYYY>/<YYYY-MM>.jsonl`（**严格复刻 年子目录 + 年-月文件** 布局，否则 dawarich Import My Data 可能不识别）。
  3. 缺列（`city/country/geodata/lonlat/...`）留空或 null；dawarich 导入后由其自身反向地理编码回填（与 §4.3 丢失列白名单一致）。
  4. 其它实体文件可不产出（dawarich 导入端按缺失容忍，与本导出 `visits` 缺失同理）。
- **本节覆盖**：原 §4.1/§4.2 中 `points/YYYY-MM.jsonl` 的路径假设**更正为** `points/<YYYY>/<YYYY-MM>.jsonl`；原 §12（3）manifest 命名待确认**关闭**（确为根级小写 `manifest.json`）。

### 13.5 三类凭证是否同一套方案（回答 prompt 新问）
**不是同一套**，geokeep 刻意用三套**相互隔离**的凭证机制：

| 凭证 | 载体 | 校验方式 | 权限 | 生命周期 |
|---|---|---|---|---|
| **登录态 Session** | Cookie（HttpOnly/SameSite=Lax）| 服务端 **HMAC-SHA256(`GEOKEEP_SECRET`, payload)** 自校验，payload=`{user_id, issued_at, nonce}` | **完整读写 + 管理** | 登录签发、登出/过期失效 |
| **上报 api_key** | URL `?api_key=` / `Authorization: Bearer` / Basic | **DB 明文等值比对**（`users.api_key`）| 仅上报写入 | 一直有效，Web 内可轮换（§7.6）|
| **分享 token** | URL `/share/{token}` | **DB 查表**（`share_links.token`）| **只读 + scope 受限** | 创建/停用/过期/删除（§5.5）|

- 为什么**不能合一**：① Session 是自校验 HMAC、放进 URL 会把「完整权限凭证」泄露到浏览器历史/Referer；② 分享 token 必须可在管理页**枚举/吊销/限定范围**，这是无状态 HMAC Session 做不到的，需 DB 背书。
- 分享 token 与 api_key **机制相近**（都是 DB 背书的不透明 bearer），但分享 token **强制只读 + scope**，api_key 仅用于上报写入；两者表与中间件均独立。

### 13.6 客户端接入配置生成器（回答 prompt 新问）
> prompt：设置页除管理 api_key 外，还要为 OwnTracks / Overland / GPSLogger 生成「可直接复制」的接入 URL 与参数。

实测印证：用户当前 `points.raw_data` 形如 `{"t":"u","bs":"false","acc":"...","alt":"...","lat":"...","lon":"...","tst":"...","vel":"...","_type":"location"}`，正是 **GPSLogger「Custom URL」以 OwnTracks 模板** 上报的特征（`bs` 取 `%ISCHARGING` 故为 `true/false`）。即 **GPSLogger 经 OwnTracks 协议命中既有 `/api/v1/owntracks/points` 路由，无须新增上报源**（依据：GPSLogger Custom URL FAQ 与 dawarich Discussion #118，见 §0/会话引用）。

设置页「客户端接入」面板**按当前 `api_key` 与 `BASE_PATH` 动态生成**以下可复制内容：
- **OwnTracks（iOS/Android 原生）**：Mode=HTTP，URL=`{scheme}://{host}{base}/api/v1/owntracks/points`，鉴权用 `?api_key=<KEY>` 或 DeviceID/UserID + Password=`<KEY>`。
- **Overland**：Receiver Endpoint URL=`{scheme}://{host}{base}/api/v1/overland/batches`，`Authorization: Bearer <KEY>`（或兜底 `?api_key=<KEY>`）。
- **GPSLogger（Custom URL，POST）**：
  - URL=`{scheme}://{host}{base}/api/v1/owntracks/points?api_key=<KEY>`
  - Method=`POST`，Content-Type=`application/json`
  - Body 模板（OwnTracks 同构，含占位符）：
    ```json
    {"_type":"location","t":"u","lat":%LAT,"lon":%LON,"tst":%TIMESTAMP,
     "acc":%ACC,"alt":%ALT,"vel":%SPD_KPH,"batt":%BATT,"bs":"%ISCHARGING"}
    ```
- **建议接口**：`GET /api/v1/clients/config`（Session）返回上述三段已填好 `api_key`/URL 的配置文本，供前端一键复制；**不新增上报路由**。
- **范围说明**：GPSLogger 在此仅作为「以 OwnTracks 模板上报的客户端」，**不**新增独立 GPSLogger 解析端点（与 §1.2 一致）；若用户希望 geokeep 原生解析 GPSLogger 自有格式，需另立需求（**待确认**）。
