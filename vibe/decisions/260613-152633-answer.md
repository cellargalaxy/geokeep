## 1. 技术栈选型

### 1.1 语言与运行时
- **Go 1.22+**
  - 依据：仓库 `.gitignore` 已含 `*.exe / *.test / go.work / coverprofile`，原始作者意图为 Go 项目；Go 1.22 起 `net/http.ServeMux` 原生支持方法与路径变量路由，省去一层框架依赖；`log/slog`（1.21+）满足结构化日志需求（需求 §8）。
  - 取舍：不选 Rust / Node。Rust 拉低生态与开发速度；Node 在「单二进制 + 嵌入式部署」要做额外打包，不符合需求 §1.3「单二进制」。

### 1.2 SQLite 驱动 —— `modernc.org/sqlite`（纯 Go）
- 依据：纯 Go 实现，无需 CGO，交叉编译至 `linux/amd64 / linux/arm64 / windows / darwin` 全部能直出静态二进制，对齐需求 §1.3「轻量部署 / 单二进制」。
- 取舍：不选 `mattn/go-sqlite3`。CGO 依赖会在 musl / 跨架构构建时引入工具链复杂度，违背单二进制原则。
- 性能权衡：纯 Go SQLite 单写吞吐约为 CGO 版本 50–70%，但需求 §8 目标仅 200 pts/s，远低于纯 Go 实现上限（公开基准约 5k+ ins/s）。**待确认**：在导入百万级 v2 archive 场景下若出现瓶颈，回退到 `mattn/go-sqlite3` 走 CGO 构建分支，留作 V2 优化选项。

### 1.3 SQL 访问层 —— `database/sql` + 手写 SQL（不引入 ORM）
- 依据：MVP 表数极少（5 张）、查询模式固定；ORM 反而增加映射成本与 raw_data JSON 处理摩擦。
- 取舍：不选 GORM / ent。GORM 对 SQLite JSON 列处理弱；ent 需 codegen 与额外学习成本。
- 工具增强：`sqlc` 可作为后续可选项生成类型安全查询；MVP 阶段直接 `database/sql` + 集中常量化 SQL 文本。

### 1.4 Schema 迁移 —— `pressly/goose`（嵌入模式）
- 依据：可作为库引入（`goose.Up`），不需要安装额外 CLI，迁移脚本通过 `embed.FS` 注入二进制；和 `modernc.org/sqlite` 兼容。
- 取舍：不选 `golang-migrate/migrate`。其 SQLite 驱动默认走 CGO，需额外构建标签处理；goose 走 `database/sql`，对驱动无偏好。
- 落地：`internal/migration/sql/0001_init.sql` 起首，文件名 `NNNN_desc.sql`，`PRAGMA user_version` 由 goose 自管。

### 1.5 HTTP 框架 —— 标准库 `net/http` + 轻中间件
- 依据：Go 1.22 ServeMux 支持 `POST /api/v1/owntracks/points` 这类带方法/路径模式的路由，覆盖需求 §7 所有 14 个路由。
- 取舍：不选 `chi / gin / echo`。中间件链很短（鉴权、限速、日志、recover），自写更可控。
- 中间件清单：`requestID → logger → recover → CORS（仅本地）→ auth(api_key | session)`。

### 1.6 鉴权
- 上报通道：`api_key` 静态字符串（查询串 / Bearer / Basic 三种解析路径，统一收口到一个解析函数）。
- Web 通道：服务端签名 Cookie Session（载荷 `user_id + issued_at + nonce`，HMAC-SHA256，密钥由 `GEOKEEP_SECRET` 注入）；不引入 `gorilla/sessions`（一个依赖换不了多少代码）。
- 密码哈希：**bcrypt（`golang.org/x/crypto/bcrypt`）**。
  - 依据：需求 §12（1）列为「MVP 倾向 bcrypt」；标准库邻接，cost=12 起。
  - 取舍：不选 argon2id；MVP 单用户场景，密码哈希性能差异不构成瓶颈，bcrypt 接口稳定。

### 1.7 流式 JSON
- 解码：`encoding/json.Decoder` 的 `Token()` API + `Decode()` 子对象。
- 编码：`encoding/json.Encoder`（导出 JSONL 时单行 `Encode`）。
- **不引入** `easyjson / jsoniter / segmentio/encoding`。理由：MVP 写盘吞吐瓶颈在 SQLite，不在 JSON 编解码；保持依赖收敛。

### 1.8 GPX 解析与生成 —— `github.com/tkrajina/gpxgo`
- 依据：唯一被广泛使用、单包内含解析与生成的 GPX 库；支持 GPX 1.0/1.1 与扩展 `<gpxtpx:speed>`。
- 取舍：不自写。GPX namespace / extension 处理坑多，自写易丢字段。

### 1.9 配置
- 来源优先级：CLI flag > 环境变量 > `geokeep.yaml`（可选） > 默认值。
- 实现：标准库 `flag` + `os.Getenv`；MVP 不引入 viper / kong。
- 关键变量：
  - `GEOKEEP_LISTEN`（默认 `:8080`）
  - `GEOKEEP_DATA_DIR`（默认 `./data`，含 `geokeep.db / imports/ / exports/ / backups/`）
  - `GEOKEEP_SECRET`（必填，启动失败兜底）
  - `GEOKEEP_OSM_TILE_URL`（默认 `https://tile.openstreetmap.org/{z}/{x}/{y}.png`；需求 §12（4）待确认是否切自建）
  - `GEOKEEP_BACKUP_CRON`（可选，cron 表达式）
  - `GEOKEEP_MAX_UPLOAD_MB`（默认 5；需求 §3.3）

### 1.10 定时任务 —— `robfig/cron/v3`
- 依据：仅用于自动备份（需求 §6.2.3），无需复杂调度。
- 取舍：不自写 ticker；cron 表达式给运维灵活度。

### 1.11 日志 —— `log/slog`（标准库）
- 依据：1.21+ 内置，结构化 JSON 输出对接日志收集器友好。
- 字段约定：`req_id / user_id / device / source / dup / latency_ms`。

### 1.12 前端 —— Vanilla JS + Leaflet（CDN-free，构建产物嵌入二进制）
- 依据：需求 §5.1「单页 SPA，嵌入到二进制」「不做热力图/聚类」；引入 Vue/React 提高构建复杂度且收益小。
- Leaflet 选 1.9.x，本地化放在 `web/static/vendor/leaflet/`，通过 Go `embed.FS` 提供。
- 模块切分：`web/static/{index.html, app.js, app.css, vendor/}`；约束 < 50KB JS（不含 Leaflet）。
- 不做构建步骤（无 npm / vite）；改前端就是改文本文件，重启二进制即可。
- **待确认**：是否需要打包压缩（`go:embed` 后再 gzip）。MVP 不做。

## 3. 数据层

### 3.1 SQLite 启动 PRAGMA
| PRAGMA | 值 | 依据 |
|---|---|---|
| `journal_mode` | `WAL` | 需求 §6.1「启用 WAL」 |
| `synchronous` | `NORMAL` | 需求 §6.1 |
| `foreign_keys` | `ON` | 需求 §2 表间外键 |
| `busy_timeout` | `5000` | 写锁等待 5s |
| `temp_store` | `MEMORY` | 流式导入临时表性能 |
| `cache_size` | `-65536`（64MB） | 平衡内存与查询性能 |

### 3.2 表 DDL 摘要（详细字段见需求 §2）

```sql
-- 0001_init.sql 关键约束
CREATE UNIQUE INDEX uq_points_user_coord_ts
    ON points(user_id, latitude, longitude, timestamp);
CREATE INDEX idx_points_user_ts ON points(user_id, timestamp);
CREATE INDEX idx_points_user_dev_ts ON points(user_id, device_id, timestamp);
```
- **去重**：`uq_points_user_coord_ts` 由数据库强制（对齐需求 §2.3「dawarich 一致」）。
- **JSON 列**：SQLite 用 `TEXT` 存 JSON，业务层用 `json.RawMessage`；不使用 `JSON1` 函数（保留可移植性，避免依赖编译标志）。

### 3.3 写并发 —— 单写者锁
- 模式：`sync.Mutex` 包裹「所有写事务入口」（`repo.WriteTx`）。
  - 依据：SQLite WAL 允许多读单写；Go 应用层加锁可避免 `database/sql` 并发触发 `SQLITE_BUSY` 重试风暴。
  - 实施：`db.WriteTx(ctx, func(tx *sql.Tx) error)`；handler 与导入/导出 goroutine 均经此入口。
- 读：不加锁，依赖 SQLite 自身 MVCC（WAL 提供）。
- 取舍：不引入 connection pool 写连接数限制（直接 `db.SetMaxOpenConns(1)` 用于写也可），改用应用层 mutex 是为了**让批量上报合并写事务**（见 §5.3）。

### 3.4 批量插入策略
- 上报通道（同步路径）：单请求内的 N 个 point（Overland 一次可达 1000）在**单一事务**内执行 N 次 `INSERT … ON CONFLICT DO NOTHING`（SQLite 3.24+ 支持 `ON CONFLICT`）。
- 导入通道：每 1000 行 commit 一次事务（兼顾持久性与吞吐）；commit 前在 `imports` 表更新 `processed/doubles`，便于前端轮询。

### 3.5 字段映射封装
所有「外部 payload → `Point`」走 `internal/ingest`：
- `ingest.MapOwnTracks(payload) -> []Point`（详见需求 §3.1）
- `ingest.MapOverland(payload) -> []Point`（详见需求 §3.2）
- `ingest.UpsertDevice(name, source) -> device_id`
- `raw_data` 一律存原始 `[]byte`（在 mapper 入口截留 reader）。

## 4. HTTP 层

### 4.1 路由表（实现需求 §7）
路由统一前缀 `/api/v1`，由 Go 1.22 ServeMux 注册：
```
POST   /api/v1/owntracks/points       handler_ingest.OwnTracks
POST   /api/v1/overland/batches       handler_ingest.Overland
POST   /api/v1/auth/login             handler_auth.Login
POST   /api/v1/auth/logout            handler_auth.Logout
GET    /api/v1/points                 handler_query.Points
GET    /api/v1/devices                handler_query.Devices
GET    /api/v1/stats/summary          handler_query.Summary
POST   /api/v1/imports                handler_io.CreateImport      (multipart)
GET    /api/v1/imports/{id}           handler_io.GetImport
POST   /api/v1/exports                handler_io.CreateExport
GET    /api/v1/exports/{id}           handler_io.GetExport
GET    /api/v1/exports/{id}/download  handler_io.DownloadExport
GET    /api/v1/backup                 handler_backup.Backup
POST   /api/v1/restore                handler_backup.Restore       (admin)
GET    /                              static.Serve
GET    /static/*                      static.Serve
```

### 4.2 中间件链
```
mux -> requestID -> slogLogger -> recover
     -> CORS (仅 dev 模式开)
     -> route 分组:
        - /api/v1/owntracks/points, /overland/batches  -> auth.APIKey
        - /api/v1/auth/login                            -> 无
        - 其它 /api/v1/**                                -> auth.Session
        - /, /static/**                                  -> 无
```

### 4.3 上报 handler 流程（同步写）
```
parse(body) -> mapper.Map* -> ingest.Save(tx)
  - 单事务内 INSERT OR IGNORE，统计 inserted/dup
  - 出错：500
  - 成功：OwnTracks -> 200 "[]"；Overland -> 200 {"result":"ok"}
```
- 依据：需求 §3.3「同步写：MVP 不引入异步队列；写库失败直接返回非 2xx，由客户端重传」；OwnTracks/Overland 文档均说明非 2xx 会进入客户端本地重传队列。
- 失败语义：**部分失败一律全 rollback**，让客户端整批重传（避免半成功后客户端去重困难）。

### 4.4 鉴权解析（上报通道）
```
apikey := query("api_key")
      || trimPrefix(header("Authorization"), "Bearer ")
      || basicAuthPassword(header("Authorization"))
      || header("X-Limit-U") // 仅作辅助识别，不做鉴权
```
- 401 行为：返回 `{"error":"unauthorized"}` + `WWW-Authenticate: Bearer` 头。
- **待确认**：OwnTracks 客户端是否会因 401 + 非空体停止重传（需求 §12（5））；MVP 先按标准方式返回，必要时改为返回 `[]` + 401。

### 4.5 文件上传
- 走 `multipart/form-data`，单字段 `file` + 可选 `format`（显式声明，避免靠扩展名猜）。
- 服务端落盘到 `data/imports/<import_id>/raw.{ext}` 后再异步解析（详见 §6）。
- 上限：`GEOKEEP_MAX_UPLOAD_MB`（默认 5MB），大文件按 `Transfer-Encoding: chunked` 流式落盘；**不**一次性 `ioutil.ReadAll`。

## 5. 上报通道详细设计

### 5.1 OwnTracks Handler 流程
```
1. read body (limit MaxUploadMB)
2. unmarshal into map[string]json.RawMessage 取 "_type"
3. switch _type:
   - "location"  -> mapper.MapOwnTracksLocation -> save
   - "transition","waypoint","status","cmd","lwt"
     -> 仅 raw_data 留档（写一条 `events` 表？MVP 不写表，直接丢弃 + log; 见 §5.4）
   - 缺省 _type -> 400
4. respond 200 "[]\n"
```

### 5.2 Overland Handler 流程
```
1. read body (limit MaxUploadMB)
2. json.Decoder 流式：
   - 期待对象，key "locations" -> 数组 Token loop
     - 每读到一个 Feature -> mapper.MapOverlandFeature -> 追加到 buffer
   - key "current"/"trip" -> json.RawMessage 暂存
3. ingest.Save(buffer, extra=raw_data.overland_extra)
4. respond 200 {"result":"ok"}
```

### 5.3 字段转换关键点（执行需求 §3.1/3.2 表）
| 字段 | 转换 |
|---|---|
| `timestamp` | OwnTracks `tst` int 直用；Overland `properties.timestamp` ISO8601 → `time.Parse(time.RFC3339)` → `Unix()` |
| `velocity` | OwnTracks `vel` int → `strconv.Itoa`；Overland `speed` m/s → `strconv.Itoa(int(round(v*3.6)))`；原值留 `raw_data` |
| `battery` | Overland `battery_level` 0–1 → `int(round(v*100))` |
| `battery_status` | Overland 字符串映射表：`unknown→0, unplugged→1, charging→2, full→3`（**待确认**与 OwnTracks `bs` 完全一致，需求 §12（2）） |
| `trigger` | OwnTracks `t` 单字符映射表（**待确认** dawarich 实际枚举，需求 §12（2）） |
| `connection` | OwnTracks `conn`：`w→0, o→1, m→2`（**待确认** dawarich 实际枚举） |
| `motion_data` | Overland `motion/activity/pauses/deferred/significant_change/desired_accuracy/locations_in_payload` 整体 JSON 化落库 |
| `inrids` / `in_regions` | OwnTracks 数组 → JSON 字符串 |

### 5.4 OwnTracks 非 location `_type` 的归宿
- MVP 决策：**仅日志 + 200 "[]"**，**不落库**。
- 依据：需求 §1.2 圈定 MVP 不实现 Visits/Regions；transition/waypoint 数据落库后无消费方。
- 风险：未来若实现 Regions，会丢失历史 transition 事件。**待确认**：是否新增 `events` 表无脑全量留档（成本低，建议加，但本方案保守不在 MVP 引入）。

### 5.5 限流与防滥用
- MVP 不在应用层限流；OSM 用量条款风险只针对前端瓦片拉取（需求 §12（4））。
- 安全：`api_key` 至少 32 字节 base64 随机；首启动 CLI 输出一次，落库后只显示尾 4 位。

## 6. 导入引擎

### 6.1 任务模型
- `POST /api/v1/imports` → 落 `imports` 行（`status=pending`）→ 落盘原文件 → 投递到 **内置 worker goroutine**（容量 1，避免并发）。
- worker 取出任务 → `status=running` → 调用对应 Parser → 流式 emit Point → 批量写库 → `status=completed/failed`。
- 进度统计：`raw_points += 1`、写库时 `processed += 1`、冲突时 `doubles += 1`；每 1000 行 commit 一次 imports 行（与 §3.4 一致）。

### 6.2 ParserInterface
```
type Parser interface {
    Parse(ctx, r io.Reader, emit func(model.Point) error) error
}
```
- 工厂：`importer.New(format string) (Parser, error)`，支持 `dawarich_v2 / dawarich_v1 / owntracks_rec / gpx / geojson`。
- 格式识别：
  - 用户上传时**显式**传 `format`；缺省时退化为按扩展名 + magic byte 启发判断（`.zip + 内含 points/*.jsonl` → v2，`data.json` → v1，`.rec` → owntracks_rec，`.gpx/.geojson` → 显然），失败则返回 400 要求显式声明。

### 6.3 各 Parser 实现要点

| Parser | 输入 | 流式策略 |
|---|---|---|
| `dawarich_v2` | `.tar.gz` / `.zip` archive | 用 `archive/tar` 或 `archive/zip` 顺序读条目；进入 `points/YYYY-MM.jsonl` 后逐行 `json.Unmarshal` 单 Point；其它子目录 (`tracks/`, `visits/`, `trips/`, `stats/`, `digests/`, `tags/`, `raw_data_archives/`) **跳过**并在 `imports.error_message` 末尾追加 `skipped: tracks,visits,trips,stats,digests,tags,raw_data_archives`（覆盖需求 §4.1 与 §2.6） |
| `dawarich_v1` | `data.json` | `json.Decoder` token 流：进 `points` 数组后 `for dec.More() { dec.Decode(&p) }`，避免 OOM（需求 §8） |
| `owntracks_rec` | `bufio.Scanner` 单行 | 行格式 `2024-01-01T00:00:00Z\tFOO\t{json}`；按 `\t` 切 3 段取第 3 段 unmarshal；与上报 mapper 共享 `MapOwnTracksLocation` |
| `gpx` | `gpxgo.ParseBytes` 不行（OOM），改用 `gpxgo.Parse(io.Reader)` 流式 | 遍历 `trk.Segments.TrackPoints` emit；`extensions` 内 `<gpxtpx:speed>` 提取为 `velocity` |
| `geojson` | `json.Decoder` token 流 | 仅接 `FeatureCollection<Point>` 与 `Feature<LineString>+properties.coordTimes`；不匹配整段返回 422（需求 §4.1「不脑补时间戳」） |

### 6.4 archive 未明确目录的容错（执行需求 §12（3））
- v2 archive 的精确目录布局未公开。Parser 实现策略：
  - 遍历所有 archive 条目，**仅当**条目路径前缀 `points/` 且后缀 `.jsonl` 时纳入；其它无脑 skipped。
  - 若 archive 内一个匹配条目都没有，返回 422 `unrecognized archive layout (expected points/*.jsonl)`，避免静默成功。
  - **后续**：实施期用一份真实 dawarich export 抓样本固化为 fixture（`testdata/dawarich_v2_sample.tar.gz`）。

### 6.5 错误与重启恢复
- 任务持久化：`imports.status / processing_started_at` 落库 + 原始文件落盘 → 进程重启后扫描 `status=running` 的任务，重置为 `pending` 再重投。
- 解析中断：单个 record parse error 不阻断整体（写 warn 日志、`error_message` 追加首 N 条样本），但 `processed` 不递增；满足「全保留 + 单条不丢整体」语义。

## 7. 导出引擎

### 7.1 任务模型
- `POST /api/v1/exports` 接收 `{name, file_format, start_at, end_at}` → 落 `exports` 行 → 投递到 worker（与导入共享单 goroutine，避免争 SQLite 写锁）。
- worker 拉数据：`SELECT ... FROM points WHERE user_id=? AND timestamp BETWEEN ? AND ? ORDER BY timestamp` 用 `*sql.Rows` 流式迭代。
- 落盘到 `data/exports/<export_id>/<name>.{ext}`，完成后 `status=completed` + 文件大小写入日志。

### 7.2 WriterInterface
```
type Writer interface {
    Open(ctx, sink io.Writer) error
    Write(ctx, p model.Point) error
    Close(ctx) error
}
```
- 工厂：`exporter.New(format) (Writer, error)`，支持 `dawarich_v2 / gpx / geojson / owntracks_json`。

### 7.3 各 Writer 实现要点

| Writer | 产物 | 关键 |
|---|---|---|
| `dawarich_v2` | `tar.gz`（包含 `points/YYYY-MM.jsonl`） | 维护一个 `(year,month) -> *tar.Writer 中的 entry`；按月切换；每写一行 `json.Encoder.Encode(point)` |
| `gpx` | 单文件 `.gpx` | 一个 `<trk>`，按 `device_id` 切 `<trkseg>`；时间戳 RFC3339 |
| `geojson` | 单文件 `.geojson` | `FeatureCollection<Point>`，`properties` 直接平铺 Point 所有列 |
| `owntracks_json` | 单文件 `.json`（行式 JSONL，每行 `_type=location` 对象） | 复用 ingest 的逆映射（同一表统一管理） |

### 7.4 字段反向映射的兼容性损耗
- 需求 §4.3 列出的「允许丢失列白名单」直接写入导出包 `MANIFEST.json`：
  ```
  {"exporter":"geokeep/v0.1","format":"dawarich_v2","missing_columns":["country","city","geodata","visit_id","track_id"]}
  ```
- 文件名固定 `MANIFEST.json`，**待确认**与 dawarich 真实 archive 是否同名冲突；目前公开文档无此文件命名提及（需求 §12（3））。备用名 `geokeep_manifest.json`。

### 7.5 下载与清理
- `GET /api/v1/exports/{id}/download` 走 `http.ServeContent`（支持 Range，便于断点续传）。
- 保留期：默认 7 天（可配 `GEOKEEP_EXPORT_TTL=168h`），cron 每天 03:00 清理 `created_at < now - TTL` 的导出物并把行的 `status` 转 `expired`。

## 8. 查询接口

### 8.1 `/api/v1/points`
- 参数：`from`、`to`（epoch 秒，必填）、`device_id`（可选，可重复）、`limit`（默认 50000，硬上限 200000）、`sample`（默认 1，按 timestamp 取模等步长抽样）。
- SQL 模式：
  ```sql
  SELECT id,timestamp,latitude,longitude,altitude,velocity,course,device_id
    FROM points
   WHERE user_id=? AND timestamp BETWEEN ? AND ?
     [AND device_id IN (...)]
   ORDER BY timestamp;
  ```
- 抽样：在 SQL 后端用 `ROW_NUMBER() OVER ... % ? = 0`（SQLite 3.25+）实现，避免拉全量到应用层。
- 响应：精简列（不返回 `raw_data / motion_data`）→ 减小 payload。

### 8.2 `/api/v1/devices`
```sql
SELECT id,name,source,
       (SELECT MAX(timestamp) FROM points WHERE device_id=devices.id) AS last_seen,
       (SELECT COUNT(1) FROM points WHERE device_id=devices.id) AS point_count
  FROM devices WHERE user_id=?;
```

### 8.3 `/api/v1/stats/summary`
- 总数：`COUNT(1)`。
- 距离：拉所有点（已按窗口与设备过滤）后在应用层走 Haversine。
  - 需求 §8 已界定 30 天 ≤ 200k 点；Haversine 1 次约 100ns，200k 点 ≈ 20ms，可接受。
  - 优化（不在 MVP）：可缓存到日聚合表。

## 10. 备份 / 恢复

### 10.1 在线热备 `GET /api/v1/backup`
- 实现：
  1. `tmpPath := data/backups/.staging-<rand>.db`
  2. 执行 `VACUUM INTO ?` 绑定 `tmpPath`（SQLite 3.27+，`modernc.org/sqlite` 已支持）
  3. `http.ServeContent` 流式吐文件，下载完成后 `os.Remove(tmpPath)`
- 命名（自动备份场景）：`geokeep-YYYYMMDD-HHMMSS.db`。
- 不阻塞写：`VACUUM INTO` 在 SQLite 内部走读快照，期间业务读写均可继续（与 WAL 配合）。

### 10.2 CLI restore `geokeep restore --from path.db`
- 步骤：
  1. 打开候选文件 → `PRAGMA integrity_check` 必须 `ok`
  2. 读 `PRAGMA user_version` ≤ 当前迁移版本（高于直接拒绝）
  3. 检查当前服务进程已停止（通过 PID 文件，`data/geokeep.pid`）
  4. 原 `geokeep.db` 改名为 `geokeep.db.bak-<ts>`
  5. 候选文件移动到 `geokeep.db`
  6. 启动期自动 goose Up（前向迁移）
- 失败回退：步骤 4 改名失败回滚 `.bak`，命令返回非 0。

### 10.3 Web restore `POST /api/v1/restore`
- 仅 admin 用户允许（MVP 单用户均视为 admin）。
- 上传 `.db` → 落到 `data/backups/upload-<ts>.db` → 同 10.2 流程（但不停服务，改用 SQLite **online backup API** 的 backstep —— **待确认** `modernc.org/sqlite` 是否暴露该 API；如不暴露，MVP 退化为：上传后写一个 `pending_restore` 标记文件，返回「服务将在重启后恢复」，下次启动期完成切换）。
- 取舍：online backup API 复杂度高，MVP 倾向「上传 + 标记文件 + 提示重启」语义，与 CLI 路径一致。

### 10.4 定时备份
- `GEOKEEP_BACKUP_CRON="0 4 * * *"`（默认空，关闭）；启用后每天 04:00 触发 `VACUUM INTO data/backups/geokeep-YYYYMMDD-040000.db`。
- 保留策略：保留最近 N 份（`GEOKEEP_BACKUP_KEEP=14`，默认 14 天）。

## 11. 配置启动与子命令

`geokeep` 二进制提供 4 个子命令：
| 子命令 | 用途 |
|---|---|
| `geokeep serve` | 默认；启动 HTTP 服务 |
| `geokeep mkuser --email --password` | 首次部署创建用户；输出 `api_key`（只此一次） |
| `geokeep rotate-key --email` | 重置 `api_key` |
| `geokeep restore --from path.db` | 离线恢复（§10.2） |
| `geokeep export --format --from --to --out path` | CLI 触发导出（绕开 Web） |

启动期顺序：
1. 加载 config（flag + env）→ 校验 `GEOKEEP_SECRET` 非空
2. 打开 SQLite → 应用 PRAGMA → goose Up
3. 若存在 `data/pending_restore` 标记 → 执行 §10.3 切换
4. 启动 cron（如配置）
5. 启动 worker goroutine（imports / exports / backup 共用一条单工 channel）
6. `http.ListenAndServe`
7. 注册 SIGTERM → 等待 worker 排空（最长 30s）→ 关 DB → 退出

## 12. 部署

### 12.1 单二进制
- `go build -trimpath -ldflags="-s -w -X version.Build=$(git rev-parse --short HEAD)" -o geokeep ./cmd/geokeep`
- 静态二进制：`CGO_ENABLED=0`（由 `modernc.org/sqlite` 保证可行）。
- 体积估算：< 30MB（Go 静态 + Leaflet 嵌入）。

### 12.2 Docker
- 多阶段：`golang:1.22-alpine` 编译 → `gcr.io/distroless/static:nonroot` 运行。
- 暴露 `8080`；卷挂 `/data`。
- 健康检查：`HEALTHCHECK CMD wget -qO- http://127.0.0.1:8080/healthz || exit 1`（新增 `/healthz` 不计入 §4.1 路由表，因属于运维口）。

### 12.3 反代建议（文档化，不打入二进制）
- Nginx / Caddy 前置 TLS；上报端建议 `client_max_body_size` ≥ `GEOKEEP_MAX_UPLOAD_MB`。

## 15. 安全要点

| 项 | 处理 |
|---|---|
| api_key 泄露 | 仅创建/轮换时显示一次；落库 bcrypt 形式？**待确认**：上报通道需明文比对，无法 bcrypt → MVP 直接存明文（与 dawarich 一致），强约束 `data/geokeep.db` 权限 0600 |
| Session 篡改 | HMAC-SHA256 + `GEOKEEP_SECRET` |
| 上传文件大小 | `http.MaxBytesReader` 包裹 body |
| 路径穿越 | `data/imports/<id>/` 落盘时 `id` 严格 int 来源，文件名服务端生成，不取用户原名 |
| CSRF | `auth.Session` 中间件验证 `Origin == Host` 或 `Sec-Fetch-Site=same-origin`；MVP 不引入 token |
| SQLi | 全部参数化 |
| XSS | 嵌入式 SPA 不渲染服务端字符串到 DOM；JSON 走 `JSON.parse` |

| 需求 §9 取舍 | 技术方案落点 |

| dawarich v2 archive 布局假设错误 | 双向迁移失败 | §6.4 容错策略 + 实施期固化真实 fixture |
| 纯 Go SQLite 在 100 万点导入吞吐不足 | 用户体验差 | 留 `mattn/go-sqlite3` CGO 编译标签备用 |
| OSM 公共瓦片用量超限 | 地图加载受限 | 文档建议自建瓦片，UI 暴露 `tile_url` 设置 |
| api_key 明文落库 | 数据库泄露即等于上报通道沦陷 | `data` 目录 0600；后续考虑 HMAC(api_key) 落库 + 客户端原文比对 |
| 单 worker 串行 import/export/backup | 大文件导入期间无法导出/备份 | 加任务队列状态可见性；用户教育「错峰」 |

## 20. 实施补充（2026-06-13，本节为最终落地依据）

> 本节对接 `260613-152633-prompt.md` 的 7 项补充约束。冲突时以本节为准。

### 20.1 性能与并发降级
- 用户给出明确边界：「**个人最多两三个设备**」。读写吞吐目标从 §8 的 200 pts/s 下调到 **≥ 50 pts/s**；导入/导出/备份共用单 worker（§7.1）的取舍仍成立。
- 影响：去除「§3.4 批量 1000 commit 一次」的优化要求，统一改为「每条 / 每 100 条 commit」均可；不会引入额外架构复杂度。

### 20.2 引入 ORM —— **GORM**（覆盖原 §1.3 决策）
- 选型：`gorm.io/gorm` + `github.com/glebarez/sqlite`（GORM 兼容驱动，**底层基于 `modernc.org/sqlite`，纯 Go**，仍保单二进制）。
  - 依据：`github.com/glebarez/sqlite` README 明文 "Pure-Go (without cgo) SQLite driver for Go that uses the GORM ORM"。
- 取舍：放弃原 §1.3「不引入 ORM」决策；引入 GORM 换取**鲁棒性**（自动迁移、字段映射、声明式约束）。
- 子驱动取舍：不选 `gorm.io/driver/sqlite`（底层 `mattn/go-sqlite3`，需要 CGO）。
- 已确认依赖（实测 `go get` 全可拉取）：
  - `gorm.io/gorm v1.31.x`
  - `github.com/glebarez/sqlite v1.11.x`
  - `github.com/tkrajina/gpxgo v1.4.x`
  - `golang.org/x/crypto/bcrypt`
- 模型层：所有 `internal/model/*.go` 用 GORM struct tag 声明字段、索引、唯一约束；`raw_data / motion_data` 用 `datatypes.JSON`（GORM 提供）或 `[]byte`+TEXT 列。
- 写并发：仍保留应用层 `sync.Mutex` 包裹 GORM 事务入口；GORM 不改变 SQLite WAL 单写者的物理事实。

### 20.3 自动建库 / 自动建表
- 实现：启动期顺序：
  1. 解析 `cfg.DataDir`，`os.MkdirAll(cfg.DataDir, 0o700)`。
  2. `gorm.Open(sqlite.Open(filepath.Join(cfg.DataDir, "geokeep.db")), ...)`：文件不存在自动新建。
  3. 应用 PRAGMA（§3.1，通过 `db.Exec`）。
  4. `db.AutoMigrate(&User{}, &Device{}, &Point{}, &Import{}, &Export{}, &Session{})`：表不存在自动创建；已存在则按 GORM 兼容规则增列（GORM 不会删列，避免数据丢失）。
- 替换原 §1.4「goose 迁移」决策（仍保留作为 V2 升级选项；MVP 用 `AutoMigrate` 满足「自动建表」）。
- 启动失败兜底：迁移异常 → fail-fast，**禁止退化到「跳过迁移启动」**（避免 schema 错位）。

### 20.4 鉴权越权防护（强化 §1.6 / §4.4）
- **绝对原则**：每一条业务 SQL **必须**包含 `WHERE user_id = ?`；通过 repository 层强制收口，禁止直接传 `id` 的「按主键查」。
  - 入口示例：`repo.GetPointByID(ctx, userID, pointID)`、`repo.GetImportByID(ctx, userID, importID)`。
  - 在单测里加「越权用例」：构造 user A 与 user B 各自的资源，断言「A 用 B 的 ID 查询返回 `ErrNotFound`，而非数据」。
- **API key 解析**：仅 `?api_key=` / `Authorization: Bearer` / Basic 三种官方路径；**`X-Limit-U/D` 仅识别设备/用户名，不参与鉴权**（与 OwnTracks 文档一致）。
- **Session**：
  - HMAC-SHA256(`GEOKEEP_SECRET`, payload) 自校验；`payload = {user_id, issued_at, nonce}`。
  - Cookie 属性：`HttpOnly; Secure(prod); SameSite=Lax; Path={BasePath}`（§20.6）。
  - **CSRF**：所有 `POST/PUT/DELETE` 强校验 `Origin` 或 `Sec-Fetch-Site=same-origin`；上报通道走 api_key，跳过 CSRF 校验。
- **管理动作隔离**：`/api/v1/restore`、`/api/v1/rotate-api-key`、`/api/v1/users` 类操作走单独中间件 `requireAdmin`（MVP 单用户即 admin，但代码层把判断写出来，避免后续扩多用户时遗漏）。
- **密码强度**：注册接口要求 ≥ 10 字符；bcrypt cost=12。
- **暴力破解**：登录失败计数 5 次后该账号 15 分钟禁登；记录到 `users.failed_attempts / locked_at`（字段在 dawarich `users` 表已有，复用）。
- **路径穿越**：导入/导出文件名服务端生成（UUID），永不取用户原名拼路径。
- **大小限制**：`http.MaxBytesReader` 包裹所有 body；上传走 `cfg.MaxUploadMB`。
- **整库审计**：所有越权敏感接口在 `slog` 打 `user_id` + `target_id` + `outcome`。

### 20.5 单测策略全面收口（覆盖 §13）
- **必测清单**（任何 PR 不达此清单则拒绝合入）：
  1. 鉴权：`apikey / session / password / 越权` 四类用例。
  2. ingest mapper：OwnTracks/Overland 每个字段一例（含 0/缺省/极值）。
  3. repository：CRUD + UNIQUE 冲突 + 越权查询。
  4. importer：每种格式（OwnTracks `.rec` / dawarich v1 / GPX / GeoJSON / dawarich v2）至少 1 个 golden fixture。
  5. exporter：每种格式 round-trip（写 → 读回 → 集合对拍）。
  6. backup：`VACUUM INTO` 后 `integrity_check` + 行数对拍。
  7. HTTP handler：每个路由 happy + 鉴权失败 + 越权 三组。
- **测试基座**：所有 repository/handler 单测在 `:memory:` SQLite 上跑（GORM 支持），fixture 走 `t.TempDir()` 拷文件。
- **CI 门禁**：`go test ./... -race -count=1`，覆盖率 ≥ 70%（暂定）。

### 20.6 子路径反代（覆盖 §4.1 / §9.1）
- 配置：`GEOKEEP_BASE_PATH`（默认 `""`，根路径）。要求**前后有限制**：必须形如 `/xxx` 或 `/xxx/yyy`，不带尾斜杠；启动期校验。
- **路由前缀**：HTTP server 把 BasePath 拼到所有路由前；ServeMux 实例位于 `internal/server`，对外是 `cfg.BasePath + "/api/v1/..."`。
- **静态文件**：
  - `index.html` 内含占位符 `{{BASE_PATH}}`；服务端**在内存中**做模板替换后返回（不修改 embed.FS 内文件本身）。
  - JS 通过 `window.__GEOKEEP_BASE__ = "{{BASE_PATH}}";` 获取，所有 `fetch(...)` 用 `BASE + "/api/v1/..."`。
  - Leaflet 静态资源也走 BasePath（embed.FS 用 `http.StripPrefix(BasePath + "/static/", ...)`）。
- **Cookie Path**：`SetCookie.Path = cfg.BasePath || "/"`，避免漏到上级路径。
- **反代示例**写入 README：
  ```
  location /xxx/ {
      proxy_pass http://127.0.0.1:8080/xxx/;
      proxy_set_header X-Forwarded-Prefix /xxx;
  }
  # 启动：GEOKEEP_BASE_PATH=/xxx ./geokeep serve
  ```
- **健康检查 `/healthz`**：不带 BasePath 前缀，便于探活。

### 20.7 Web 页面化初始化（覆盖 §11 `mkuser` 子命令）
- 流程：
  1. 任意未登录请求命中 Web → 服务端通过 `repo.UserCount() == 0` 判断「未初始化」→ 前端展示「**初始化向导**」页（注册首用户）。
  2. `POST /api/v1/setup`：仅当 `UserCount() == 0` 时接受；创建 `User{ admin: true }`，**响应体一次性返回 `api_key`**（之后只显示尾 4 位）。
  3. 初始化完成后跳转 `/login`。
- 安全：`setup` 接口在 user_count > 0 时返回 410 Gone，避免反复触发。
- 同时**保留** CLI `geokeep mkuser`（可选），但 Docker 用户无需进容器执行任何命令。

### 20.8 Docker 部署 + GitHub Actions
- **Dockerfile**：
  - 多阶段，`golang:1.22-alpine` 编译（`CGO_ENABLED=0`）→ `gcr.io/distroless/static:nonroot` 运行。
  - 端口 `8080`；数据卷 `/data`；env `GEOKEEP_DATA_DIR=/data`、`GEOKEEP_LISTEN=:8080`、`GEOKEEP_SECRET=<必须由用户传入>`。
  - 不内置 secret；用户必须在 `docker run -e GEOKEEP_SECRET=...`。
- **GitHub Actions**：`.github/workflows/docker.yml`
  - 触发：`push: branches: ['**']`（任意分支 push）。
  - 镜像 tag 策略：
    - `ghcr.io/<owner>/geokeep:<branch>` —— 每分支最新（含 `main`）。
    - `ghcr.io/<owner>/geokeep:<branch>-<short-sha>` —— 不可变标签便于回滚。
  - 步骤：checkout → setup-buildx → login ghcr → build & push（`linux/amd64,linux/arm64` 多架构，仅 main 分支跨架构以省 CI 时间，其他分支只构 amd64）。
- **取舍**：不内置 tag-on-release（用户未要求），任意 push 即出新镜像。

### 20.9 api_key 在线轮换
- `POST /api/v1/api-key/rotate`（session 鉴权）→ 生成新 key（32 字节 url-safe base64）→ 替换 `users.api_key` → 响应体返回新 key（仅此一次明文）。
- 旧 key 立即失效（同一字段被覆盖）；客户端需重新配置。
- **取舍**：不维护历史 key 表 / 不做灰度过渡。理由：用户场景为单用户最多 3 设备，重新配置成本极低。
- UI：登录后在「设置」页提供「显示 api_key 尾 4 位 + Rotate 按钮」，确认弹窗后调用。
- 审计：`slog` 打 `event="api_key_rotate" user_id=...`。
